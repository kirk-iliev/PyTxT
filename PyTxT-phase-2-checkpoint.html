Phase 1 — what's shipped Complete

Phase 1 established the architectural skeleton — soft IOC + CA client + FastAPI + browser frontend on a single asyncio loop — and validated it round-trip with a trivial PING command. No beam-physics yet; every architectural seam that phases 2–5 depend on is now in place and proven.

What it proves

  • A caproto soft IOC publishes app state as PVs that external CA clients can subscribe to.
  • A FastAPI service serves REST + a WS bridge that exposes those same PVs to the browser.
  • A browser observes state via WS and issues commands via REST POST.
  • An external agent issues the same commands via CA writes, with bit-identical effect.
  • Every subsystem has a clearly bounded home — phases 2–5 add code without restructuring.

Key invariants from phase 1 that phase 2 inherits

InvariantHow it's enforced
Every CMD has identical effect via CA-write and REST-POST Both routes call the same handler function. Enforced by import graph, not review discipline. tests/integration/test_parity.py is the load-bearing test.
AppState is the single in-process source of truth Handlers mutate AppState; the IOC publisher mirrors changes outward to PVs. No handler writes to a PV directly.
Domain logic has zero I/O dependencies pytxt/domain/ imports only numpy/scipy. No caproto, no FastAPI, no asyncio. Testable in milliseconds.
Dev IOC cannot accidentally publish prod PV names Belt-and-suspenders: PV-prefix (OSPREY:TEST:TXT: vs TxT:) AND port (59064 vs 5064). Both wrong simultaneously to collide.

Phase 2 — what we're about to build Spec'd, ready

Value moment

The operator (or an Osprey agent) issues ACQUIRE. Within ~2 seconds the browser renders a ring trajectory plot showing each of ~120 storage-ring BPMs' first-turn X and Y position. The same data is published as EPICS PVs so any external CA subscriber sees identical results.

This is the first phase where the app does something beam-physics-meaningful. Phase 2 fills the previously-empty pytxt/domain/ and pytxt/ca_client/ packages without restructuring anything from phase 1.

REST endpoints added

MethodPathPurpose

Data flow during one ACQUIRE

1
Trigger arrives: CA write to CMD:ACQUIRE OR POST /cmd/acquire. Both call the same function: handle_acquire(state, reader).
2
Concurrency check: if acquire_in_flight → reject. CA path: alarm severity. REST path: 409 Conflict.
3
Set acquire_in_flight=True, status=ACQUIRING. IOC publisher mirrors to STATE PVs.
4
Parallel CA reads: asyncio.gather over ~480 PVs (120 BPMs × 4 channels). Persistent caproto Context — connections already warm. Per-PV timeout 2 s. Missing/bad BPMs → None in the dict.
5
Domain analysis (pure numpy, no I/O): per BPM, detect_injection_turn(sum_wf) → int; extract X/Y at that index; nm → mm; None → NaN/−1.
6
Update AppState atomically: last_acquire = full result, last_acquire_raws = raw snapshot (~144 MB in memory, overwritten each ACQUIRE).
7
IOC publisher fires CA monitor events for all RESULT:BPM:* and STATE:LAST_ACQUIRE_* PVs.
8
finally: clear acquire_in_flight. IOC publisher mirrors. (Even on exception this runs.)
9
Handler returns AcquireResponse. REST returns it as JSON; CA returns success to writer.
10
WS bridge propagates CA monitor events to browser. trajectory.js re-renders Canvas plot.

Expected end-to-end latency: <3 seconds, dominated by parallel CA reads. Domain analysis on 100k-sample arrays is microseconds. Concurrent ACQUIRE attempts are rejected cleanly (not queued).

Testing strategy

  • Unit: 4 → 7 files (+ injection-turn, first-turn-extract, handlers-acquire)
  • Integration: 7 → 11 files; parity test +1 parametrize row
  • E2E: 2 → 3 files (+ trajectory.spec.js)

New infrastructure

  • Fake BPM IOC fixture — caproto-based; serves wfr:TBT:{c0,c1,c3,armed} with deterministic data (known peak at sample 1370). Supports fault injection (timeout, disconnect).
  • Tests parametrize over N ∈ {1, 5, 120}.
  • No tests against real beam (manual validation only).

Open architectural questions Track but not blocking

QuestionStatusWhen it bites
Osprey integration shape — HTTP/REST vs MCP vs CA direct vs all three? Designing for all three to be cheap to add. Phase 2's surface satisfies this. When Osprey wants to actually invoke PyTxT in anger.
srinjectoneshot + steppv MML black boxes Deferred. Hides PVs needed for injection trigger + CM step. Phase 4. Read MATLAB sources or extract underlying PV writes directly.
WS message type discriminator Deferred from phase-1 code review. Implicit field-presence works for ≤2 message types. When phase 3 adds more WS message kinds OR client logic gets complex.
Whether bpm_prefixes is runtime-reloadable Decision: load-once at startup. Requires restart to change. If BPM topology changes more often than expected (unlikely — once per few years).
Phase-1 deploy + Phoebus-subscribe validation Decided 2026-05-18: deferred as premature. Phase-1 PVs (heartbeat + ping counter) test nothing meaningful. Real deploy validation happens with phase 2's BPM read path. End of phase 2 (M4 or just after).
