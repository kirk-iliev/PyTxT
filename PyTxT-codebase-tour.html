PyTxT — Codebase Tour

A guided walkthrough of every package and every file. Built for readers who want to understand the shape of the codebase without diving into the source.

What is PyTxT?

PyTxT is a service that runs at the Advanced Light Source (ALS) particle accelerator. When operators inject a fresh bunch of electrons into the storage ring, ~120 sensors called BPMs (Beam Position Monitors) record the beam's position turn-by-turn — tens of thousands of samples each. PyTxT pulls those raw waveforms, finds the "injection turn" in each one, and publishes the per-BPM positions so that operators (and automation agents) can see the beam's trajectory and correct it.

It's a port of an older MATLAB GUI (TxT_GUI.mlapp) into a Python service with a browser UI. The design is deliberately agent-callable first: every meaningful action has both a REST endpoint and an EPICS PV, so an automation agent can drive it without going through the browser.

① Agent-firstEvery command lives at both a REST route and a CA-write PV. Browser uses the same surface.
② PVs are stateApp state is published as EPICS PVs via an embedded soft IOC. Browser is just another CA client.
③ REST fills gapsBulk transfers, file uploads, and the WS→CA bridge go through REST/WS — not PVs.
④ Pure domainAnalysis code (domain/) has zero I/O. Testable with numpy in milliseconds.

Architecture at a glance

Data enters from the left (real BPMs on the storage-ring network), flows through the Python service in the center, and exits on the right to two kinds of consumer: the browser UI and any external EPICS client (Phoebus, archiver, automation agents). Click any box to jump to that part of the tour.

pytxt/

Top of the application package — entry points and wiring.

Where the program starts and where every subsystem is constructed. Two files only at this level; everything else lives in subpackages below.

pytxt/composition.py composition root Reads config, builds every subsystem, hands them to one asyncio loop.

Plain English

This is the "main" of the program. It reads settings, loads the list of BPMs from disk, creates the in-memory state, builds the CA client (BpmReader), the soft IOC, and the FastAPI app — then starts them all on a single asyncio event loop together with a 1 Hz heartbeat task.

Why it exists

To keep object construction in one place. Every subsystem accepts its collaborators as arguments — no module reaches out to fetch a global — so swapping one out for testing is straightforward.

Key pieces

  • main() — async entry point; assembles everything and asyncio.gather()s the IOC, the uvicorn API server, the heartbeat loop, and a delayed reader-startup.
  • _resolve_version() — reads the installed package version (or falls back to 0.0.0+dev).
  • Reader startup is delayed by 1 s so the IOC has time to come up first — caproto's name resolution sees a stable network state that way.

pytxt/config/

Where settings come from.

Environment-variable-driven settings plus the static list of BPMs the app should poll. Defaults are dev-safe: out of the box the service runs in a sandboxed PV namespace so it cannot collide with real ALS hardware.

pytxt/config/settings.py config All knobs the operator can turn, with safe defaults.

Plain English

A Pydantic settings class. Every field becomes an env var (prefixed PYTXT_). Defaults boot in dev mode — PV prefix OSPREY:TEST:TXT:, ports 59064/59065 — so running it on a laptop is harmless.

Why it exists

Configuration in one typed place. The two safety rules are enforced here, not in comments: (1) the PV prefix must end with :, and (2) unknown PYTXT_* env vars are rejected at startup so a typo doesn't silently use the default.

Key pieces

  • Settings — the dataclass; fields cover the PV namespace, IOC ports, API host/port, log level, heartbeat cadence, and BPM read timeout.
  • _reject_unknown_pytxt_env_vars() — typo guard. PYTXT_BMP_PORT (instead of BPM) raises at startup, not later.
  • _prefix_must_end_with_colon() — prevents the most common config bug.

bpm_prefixes.txt composition.py
pytxt/config/bpm_prefixes.txt data Plain-text list of ~120 ALS storage-ring BPM PV prefixes.

Plain English

One BPM prefix per line, e.g. SR01C:BPM1. Comments at the top of the file record exactly how the list was extracted from MATLAB so it can be regenerated if the ring layout changes.

Why it exists

No upstream system stores this catalog as static data — MATLAB derives it live every time. Committing a one-time snapshot avoids the Python service having to talk to MATLAB at startup.

pytxt/state/

The single in-memory source of truth.

One typed dataclass holds everything the app knows about itself right now. Other subsystems subscribe to changes on individual fields, so an update to last_acquire automatically fans out to whoever cares (the IOC, in particular, mirrors it onto PVs).

pytxt/state/app_state.py state Dataclass + async pub/sub on a per-field basis.

Plain English

An AppState instance is created at startup and passed into every subsystem. You read fields directly (state.heartbeat) and you write them via await state.update(field=value, ...). The update call notifies any listener that was registered for that field.

Why it exists

So the IOC, the REST routes, the WS bridge, and the handlers don't have to pass shared state around as method arguments. It's also the place where "what changed" produces a notification — which is how PV writes get triggered.

Key pieces

  • update(**changes) — two-pass: apply all field changes first, then fire listeners. Guarantees that a listener reading other fields always sees the post-update state.
  • subscribe(field_name, callback) — register an async callback to fire on change.
  • uptime_s property — derived from started_at, computed live.
  • Stores Phase-1 fields (heartbeat, ping_count, …) and Phase-2 fields (acquire_in_flight, last_acquire, last_acquire_raws).
  • Bad field names raise AttributeError at the mutation site — typos can't quietly create new attributes.

For the curious — the two-pass update Why two passes? A listener for field A might read field B. If we notified mid-iteration, the listener could see a half-applied snapshot. 
# Pass 1: apply all changes

    ...

    actually_changed.append((k, v))

# Pass 2: notify (state is now fully consistent)
for k, v in actually_changed:
    for cb in self._listeners.get(k, []):
        await cb(v)

pytxt/ca_client/

Talks to the real BPMs over EPICS Channel Access.

A thin persistent client built on caproto. Opens connections at startup, holds them for the life of the process, and on each ACQUIRE fetches every BPM's four channels in parallel.

pytxt/ca_client/bpm_reader.py CA client Reads ~120 BPMs × 4 channels in parallel; ~500 reads per ACQUIRE.

Plain English

For each configured BPM, the reader holds four PV objects: c0 (X waveform), c1 (Y waveform), c3 (sum signal — used for injection-turn detection), and armed (a status flag). When read_all() is called, all those reads run concurrently via asyncio.gather.

Why it exists

Opening fresh CA channels on every acquire would take ~5–10 seconds and add a lot of broadcast traffic. A persistent client makes ACQUIRE finish in <3 s for the full ring of ~107 BPMs.

Key pieces

  • BpmReader.start() — opens the caproto Context and resolves all PV names. Sets _started = True only on success. Failures bubble up loudly rather than silently returning all-None reads.
  • BpmReader.read_all() — returns a dict mapping each BPM prefix to a RawBPM (or None if that BPM failed/timed out). Result preserves the configured BPM order.
  • Shape check: a waveform that isn't exactly (100000,) is treated as a read failure (logged, returns None).
  • BpmReader.stop() — idempotent; closes the caproto Context cleanly.

Talks to

caproto.asyncio.client ~120 BPM IOCs handlers/acquire.py domain/types.py (RawBPM)
For the curious — the parallel-read inner loop One coroutine per BPM. asyncio.gather turns them into one fan-out. 
c0_r, c1_r, c3_r, armed_r = await asyncio.gather(
    asyncio.wait_for(channels["c0"].read(),    timeout=self._timeout),
    asyncio.wait_for(channels["c1"].read(),    timeout=self._timeout),
    asyncio.wait_for(channels["c3"].read(),    timeout=self._timeout),

)

pytxt/domain/

Pure analysis. No I/O, no frameworks, just numpy.

Everything in this package can be unit-tested in milliseconds with hand-built numpy arrays. There are no asyncio imports, no caproto, no FastAPI — those would tie the analysis to a particular delivery mechanism.

pytxt/domain/types.py domain The two dataclasses that move between layers.

Plain English

Two frozen dataclasses: RawBPM (one BPM's just-read waveforms) and FirstTurnResult (the analyzed per-BPM positions at the injection turn).

Key pieces

  • RawBPM — fields are the three int32 waveforms (X/Y/sum, shape (100000,), units nm), an armed flag, and a read timestamp.
  • FirstTurnResult — three float64 arrays of per-BPM positions (NaN where the BPM failed), an int32 array of detected injection-turn indices (−1 where failed), and a list of failed BPM names.

Talks to

ca_client/bpm_reader.py domain/first_turn_extract.py handlers/acquire.py ioc/server.py
pytxt/domain/injection_turn.py domain Finds the sample index where the beam first arrives at this BPM.

Plain English

Given one BPM's sum-signal waveform (100,000 samples — basically the total intensity), this function locates the moment the beam first hit by looking for the largest jump from one sample to the next.

Why it exists

BPMs around the ring see the bunch arrive at slightly different times (it's traveling), so injection-turn detection has to be per-BPM, not global. Direct port of the MATLAB algorithm from SCexp_ALS_readoutBPMs.m.

Key pieces

  • detect_injection_turn(sum_waveform) — argmax(diff)) + 1. The "+1" converts a diff index into a turn index.
  • Falls back to 1370 if the detected index is outside [100, 4500] — preserves the MATLAB behavior for noisy/inactive BPMs.
  • Doesn't re-validate input shape — upstream (BpmReader) guarantees (100000,) or returns None.

Talks to

numpy domain/first_turn_extract.py
pytxt/domain/first_turn_extract.py domain Turns a dict of raw waveforms into aligned per-BPM result arrays.

Plain English

For every BPM in the dict, find its injection turn (via detect_injection_turn), then sample the X/Y/sum waveforms at that index. Converts X and Y from nanometers to millimeters (divide by 1e6) — matches the MATLAB reference.

Key pieces

  • extract_first_turn(raws) — returns a FirstTurnResult; failed BPMs (raws[prefix] is None) get NaN positions and a turn index of −1, plus their name in failed_bpm_names.
  • Dict-insertion order defines BPM index; it's the caller's job (BpmReader) to preserve the configured order.

pytxt/handlers/

Canonical command logic — one function per CMD.

For every command PyTxT supports, exactly one function lives here. Both the REST route and the IOC's CMD-PV write call it. This is the structural guarantee that REST and CA paths can't drift apart.

pytxt/handlers/ping.py handler "I'm here" — increments a counter and records a timestamp.

Plain English

Updates two fields on AppState: ping_count += 1 and last_ping_at = now. That's it.

Why it exists

It's the trivial smoke-test command that proved end-to-end parity worked in Phase 1: REST POST /api/v1/cmd/ping and a CA write to CMD:PING must increment the counter identically.

Talks to

api/routes/cmd.py ioc/pvs.py (CMD:PING putter) state/app_state.py
pytxt/handlers/acquire.py handler Orchestrates one full BPM acquisition end to end.

Plain English

Sets acquire_in_flight = True, reads every BPM in parallel via the CA client, runs the domain layer to extract per-BPM first-turn positions, writes the results onto AppState (which makes the IOC publish them and the WS bridge fan them out), then clears the in-flight flag.

Why it exists

Same function whether called from REST or from a CA-write to CMD:ACQUIRE. Concurrent attempts raise AcquisitionInFlightError, which the REST route turns into HTTP 409 and the CA path swallows (the in-flight PV already tells the caller).

Key pieces

  • handle_acquire(state, reader) — the orchestration coroutine.
  • _classify(ok, fail) — 0 ok → FAILED; any fail → PARTIAL; all ok → OK.
  • _median_injection_turn() — robust summary across all successful BPMs.
  • try/except/finally guarantees acquire_in_flight is always cleared, even when the reader raises.
  • On unexpected failure, publishes a FAILED LastAcquireResult with the error name in fail_reason so operators can see why, not just that something broke.

pytxt/ioc/

The embedded soft IOC. Publishes app state as EPICS PVs.

An EPICS "IOC" (Input/Output Controller) is just a server that publishes named values. Caproto lets us run one in-process. Every meaningful piece of AppState is mirrored onto a PV here, which makes the browser, Phoebus, archivers, and automation agents able to subscribe to PyTxT in the exact same way they subscribe to any other ALS hardware.

pytxt/ioc/pvs.py IOC Declares every PV the IOC publishes (HEALTH, STATE, RESULT, CMD).

Plain English

One caproto PVGroup class. Each pvproperty declaration is a single named value: name, type, max length (for arrays), and a human-readable description that goes into the PV's .DESC field — discoverable to anyone introspecting the IOC.

Why it exists

So agents can list and read every PyTxT signal without secret knowledge. The PV names follow a deliberate three-prefix convention: HEALTH:* for liveness, STATE:* for app state, RESULT:* for analysis output, CMD:* for write-only command triggers.

Key pieces

  • HEALTH:HEARTBEAT, HEALTH:UPTIME_S — liveness.
  • STATE:VERSION, STATE:LAST_PING_AT, STATE:PING_COUNT, STATE:ACQUIRE_IN_FLIGHT, STATE:LAST_ACQUIRE_* — current app state.
  • RESULT:BPM:X_FIRST_TURN, RESULT:BPM:Y_FIRST_TURN, RESULT:BPM:SUM_FIRST_TURN, RESULT:BPM:INJECTION_TURN, RESULT:BPM:NAMES — fixed-length 128 arrays (NaN for unused slots).
  • CMD:PING, CMD:ACQUIRE — writable. Writing any value dispatches to the matching handler. The value is ignored — these are pure triggers.
  • @cmd_ping.putter / @cmd_acquire.putter decorators — wire the CA writes to the shared handle_ping / handle_acquire functions.

Talks to

handlers/ping.py handlers/acquire.py ioc/server.py caproto.server
pytxt/ioc/server.py IOC Lifecycle wrapper: binds AppState changes to PV writes; runs forever.

Plain English

Owns the PyTxTPVGroup. On startup it subscribes to AppState changes — when a tracked field changes, this file pushes the new value onto the matching PV. That's the one-way AppState → PV data flow.

Why it exists

To decouple "what the app knows" (AppState) from "what's published over the network" (the PV namespace). Subsystems that mutate state never have to remember to also touch PVs.

Key pieces

  • PyTxTIOC.__init__ — builds the PVGroup and calls _bind_state_changes().
  • _bind_state_changes() — wires per-field listeners. Simple 1:1 fields use a small loop; last_acquire is special (it fans out to 6 scalar PVs plus 4 waveform PVs).
  • _publish_last_acquire() — re-runs extract_first_turn with prefix-aligned input so the published waveform indices line up with RESULT:BPM:NAMES.
  • run() — boots caproto's Context. Has careful env-var juggling: EPICS_CA_SERVER_PORT must be set during Context construction (so the server binds the right port) but restored afterward (so the in-process BpmReader CA client can search the ring on the standard port).
  • _force_close_context_sockets() — workaround for a caproto shutdown hang on multi-interface Linux hosts (appsdev2). Closes raw sockets so any blocked UDP recv unblocks and the task can actually cancel.
  • wait_until_running() — convenience for tests; returns once the startup hook has pushed initial PV values.

    pv = getattr(self.pvgroup, pv_attr)

Soft IOC — published PV catalog

The IOC is a caproto-based EPICS server running inside the same Python process as everything else. It listens on the configured port (dev: 59064, prod: 5064 standard EPICS) and publishes 19 named values that together expose the entire observable state of PyTxT. Anyone on the network who speaks Channel Access — Phoebus, the archiver, an automation agent, or the browser via the WS bridge — can subscribe to any of these without special knowledge of PyTxT.

Names are prefixed by the configured namespace: OSPREY:TEST:TXT: in dev, TxT: in prod. So HEALTH:HEARTBEAT becomes TxT:HEALTH:HEARTBEAT on the deployed appsdev2 instance. Tables below show the bare name; prepend the prefix to use it.

HEALTH:* liveness — answers "is PyTxT alive right now?"

Two PVs that any monitoring system can poll. HEARTBEAT increments every 1 s; if it stops, the process is wedged or gone. UPTIME_S tells you how long the current run has been up.

PV nameTypeDescription
HEALTH:HEARTBEAT R int Liveness counter. Increments by 1 every PYTXT_HEARTBEAT_INTERVAL_S seconds (default 1.0). A stalled or dead process shows a frozen value. Subscribers can detect liveness with no polling.
HEALTH:UPTIME_S R float Seconds since the process started. Pushed once per heartbeat tick (same cadence as HEARTBEAT). Reset to ~0 by any restart, so a sudden drop signals a recent restart.

STATE:* current application state — version, ping activity, acquisition lifecycle

These mirror fields of AppState. STATE:VERSION, LAST_PING_AT, PING_COUNT are Phase-1 holdovers from the ping smoke test. STATE:ACQUIRE_* and the seven LAST_ACQUIRE_* PVs together describe the most recent acquisition's outcome.

PV nameTypeDescription
STATE:VERSION R string Semantic version of the running app (e.g. 0.1.0 or 0.0.0+dev). Set once at startup; doesn't change while the process runs.
STATE:LAST_PING_AT R string ISO-8601 UTC timestamp of the most recent ping. Empty before the first ping.
STATE:PING_COUNT R int Total pings received since startup. Pure counter — resets only at restart.
STATE:ACQUIRE_IN_FLIGHT R int (0/1) 1 while an acquisition is in progress; 0 otherwise. The handler also uses this to reject concurrent CMD:ACQUIRE writes. Subscribe to this to know when to expect new RESULT:* values.
STATE:LAST_ACQUIRE_STATUS R int (enum) Lifecycle status of the most recent ACQUIRE. 0=NEVER (no acquire yet), 1=ACQUIRING (in progress), 2=OK (all BPMs returned valid data), 3=PARTIAL (some BPMs failed), 4=FAILED (no BPMs returned data, or the orchestrator itself raised).
STATE:LAST_ACQUIRE_OK_COUNT R int Number of BPMs that returned valid waveforms on the most recent ACQUIRE.
STATE:LAST_ACQUIRE_FAIL_COUNT R int Number of BPMs that timed out or returned invalid data. OK_COUNT + FAIL_COUNT equals the configured BPM count.
STATE:LAST_ACQUIRE_TIMESTAMP R string ISO-8601 UTC timestamp of when the most recent acquisition completed (success, partial, or failed). Empty before the first acquire.
STATE:LAST_ACQUIRE_FAIL_REASON R string Short error message when STATUS=FAILED (e.g. RuntimeError: BpmReader.read_all() called before start()). Empty otherwise. Lets agents diagnose without scraping logs.
STATE:LAST_ACQUIRE_FAILED_BPM_NAMES R string[128] Names of the BPMs that failed on the most recent ACQUIRE, padded to length 128 with empty strings. The first FAIL_COUNT entries are the real names; the rest are "".

RESULT:BPM:* analyzed per-BPM positions from the most recent ACQUIRE

Five fixed-length waveforms of size 128 (covers ~120 ALS BPMs with headroom). Indices across all five align — index i in every waveform refers to the BPM whose name is at index i of NAMES. NaN (for floats) or −1 (for ints) marks unused slots and failed BPMs.

PV nameTypeDescription
RESULT:BPM:NAMES R string[128] Canonical BPM prefix for each index, e.g. SR01C:BPM1 at index 0, SR01C:BPM3 at index 1, … Static after startup — set once from the BPM-prefixes file. The "index → BPM" lookup table for every other RESULT waveform.
RESULT:BPM:X_FIRST_TURN R float[128] Per-BPM X position (in mm) at the detected injection turn. NaN at indices of failed BPMs and at unused slots beyond the configured BPM count.
RESULT:BPM:Y_FIRST_TURN R float[128] Per-BPM Y position (in mm) at the detected injection turn. Same indexing and NaN convention as X.
RESULT:BPM:SUM_FIRST_TURN R float[128] Per-BPM sum-signal value (arbitrary units, raw int from the BPM IOC cast to float) at the detected injection turn. Useful for charge/intensity diagnostics. NaN for failed BPMs.
RESULT:BPM:INJECTION_TURN R int[128] Per-BPM detected injection-turn sample index in the 100,000-sample TBT waveform. −1 for failed BPMs and unused slots. Usually clusters tightly across the ring; the median is also published as STATE:LAST_ACQUIRE_* for monitoring.

CMD:* write-only triggers — the only PVs you can write to

Every other PV is read-only. These two accept writes; the written value is ignored (they're pure triggers). Each one dispatches to the matching handler in handlers/ — exactly the same function that the equivalent POST /api/v1/cmd/* REST route calls. That structural sharing is what guarantees agentic parity between CA and REST.

PV nameTypeDescription
CMD:PING W int Write any value to register a ping. The value is ignored. Effect: STATE:PING_COUNT increments by 1 and STATE:LAST_PING_AT updates to the current UTC timestamp. Identical effect to POST /api/v1/cmd/ping.
CMD:ACQUIRE W int Write any value to trigger a full BPM acquisition. The value is ignored. Effect: reads every configured BPM in parallel, analyzes the waveforms, publishes new STATE:LAST_ACQUIRE_* and RESULT:BPM:* values. If an acquisition is already in flight, the CA write silently succeeds — the caller can see STATE:ACQUIRE_IN_FLIGHT=1 and know to wait. Identical effect to POST /api/v1/cmd/acquire.

Boot order — what's defined vs. populated when

  1. caproto's Context is constructed with the full pvdb from pvs.py — every PV exists from second zero with default values (0, "", or zero-filled arrays).
  2. The startup hook pushes Phase-1 initial values: STATE:VERSION, HEALTH:HEARTBEAT=0, UPTIME_S=0, etc.
  3. The startup hook pushes Phase-2 initial values: RESULT:BPM:NAMES is populated from the BPM-prefixes file (this is the only RESULT PV that's meaningful before the first ACQUIRE), and STATE:LAST_ACQUIRE_STATUS is set to NEVER=0.
  4. A 1 Hz heartbeat task starts pushing HEALTH:HEARTBEAT and HEALTH:UPTIME_S.
  5. ~1 s later, BpmReader.start() opens its CA client and resolves all ~120 BPM PV names. The IOC is now ready to handle CMD:ACQUIRE.
  6. Every subsequent change to a tracked AppState field automatically pushes onto its matching PV via the listeners registered in ioc/server.py.

Subscribe from Python (caproto)

from caproto.asyncio.client import Context

ctx = Context()
(pv,) = await ctx.get_pvs("TxT:STATE:LAST_ACQUIRE_STATUS")
async for response in pv.subscribe():
    print("status:", response.data[0])

This is exactly what the WS bridge does internally — one task per PV.

from caproto.asyncio.client import Context

pytxt/api/ws_bridge.py WS bridge Each browser tab gets a WebSocket; the bridge re-subscribes to PVs as a caproto client and forwards updates as JSON.

Plain English

The browser opens a WebSocket to /api/v1/pvs, sends messages like {"action": "subscribe", "pvs": ["TxT:HEALTH:HEARTBEAT"]}, and gets back JSON value updates as the PVs change.

Why it exists

Browsers can't speak EPICS Channel Access (UDP broadcast + binary protocol). But routing browser updates through CA — rather than directly through AppState — preserves the "browser is just another CA client" rule. The browser sees identical type coercions and update semantics as Phoebus.

Key pieces

  • pvs_ws() — per-connection handler. State machine: accept → receive subscribe messages → fan out updates until disconnect → tear down all subscriptions.
  • _forward_pv(pv_name) — one task per PV subscription. Reads the current value, sends it, then enters the caproto subscription loop and forwards each update.
  • _coerce_value() — single-element arrays unwrap to scalars; bytes decode to UTF-8 strings; numpy scalars unwrap to Python primitives. The result is JSON-safe.
  • Bad PV names time out after 2 s and the browser gets a WSError message — no silent hangs.
  • On disconnect: every per-PV task is cancelled and awaited; no leaked subscriptions.

pytxt/api/schemas/

Pydantic models — the typed shape of every REST and WS message.

Schemas are the load-bearing documentation. Every field has a description=…, every enum has explicit values, every snake_case name is unambiguous. FastAPI uses these to auto-generate the OpenAPI spec, so agents that read the spec can discover everything.

tests/

Three tiers: unit · integration · e2e. ~80+ tests today.

Unit tests cover pure logic with no I/O. Integration tests spin up a real caproto IOC on an ephemeral port and exercise the full Python stack. E2E tests drive the actual browser via Playwright.

tests/conftest.py test infra Pins caproto to localhost + a per-session ephemeral port — never collides with real IOCs.

Plain English

Picks a random free TCP port at session start, then sets every relevant EPICS_* env var to point at 127.0.0.1 on that port. The autouse fixture ensures every integration test inherits the same isolation — nothing can leak onto the ALS network even if a test is run on a control-room host.

tests/fixtures/fake_bpm_ioc.py test infra Spins up N fake BPM IOCs in-process for integration tests.

Plain English

A pytest fixture that lets a test say "give me 107 fake BPMs" (or "give me these specific prefixes") and get a running caproto IOC that synthesizes deterministic TBT waveforms. The sum signal has a clean step at sample 1370 so the injection-turn detector produces predictable results; X/Y have a small per-BPM offset so the trajectory plot has structure to look at.

Key pieces

  • _synthesize_waveforms(bpm_index) — returns deterministic (x, y, sum) arrays seeded from the BPM index.
  • fake_bpm_ioc fixture — parametrize via @pytest.mark.parametrize('fake_bpm_ioc', [N], indirect=True).
  • Teardown force-closes raw sockets before cancelling, same shutdown trick as ioc/server.py.

tests/integration/test_parity.py load-bearing The agentic-parity invariant. If this test ever goes red, the project is broken.

Plain English

For every command PyTxT exposes, this test issues it once via CA write and once via REST POST, then compares the resulting AppState projection field-by-field. Anything that differs is a violation of the agent-first design principle.

Why it exists

Because the structural guarantee — "REST and CA call the same handler function" — can be quietly broken if someone adds a new code path. This test catches that automatically.
