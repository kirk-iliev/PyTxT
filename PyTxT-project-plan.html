PyTxT — Turn-by-Turn Analysis App

Project plan — port of MATLAB TxT_GUI.mlapp to a web frontend + Python backend + soft IOC. Handoff to Kirk.

Goal

Port ~/physbase/users/thellert/automated_startup/GUI/TxT_GUI.mlapp (MATLAB SC-toolkit GUI) to a clean web frontend + Python backend + soft IOC, mirroring the PyBeamViewer architecture with one key upgrade: application state is published as EPICS PVs, not just exposed via REST. Aspirational demo of an AI-native ALS control-room app.

Why an IOC, not just REST: in a future control room, the analysis app's internal state (fit results, reference trajectory, BBA offsets, "tracking on/off") is itself control-system data. Publishing it as PVs means Phoebus screens, archivers, the logbook, and other AI agents can subscribe to the app the same way they subscribe to a magnet IOC — no special web API knowledge required. The web frontend then becomes "just another CA client that happens to render images." PyBeamViewer is REST/WS only, so its state lives in process memory; the TxT app should not repeat that.

Scope locked

  • In: TxT_GUI workflow — first-turn threading visualization, BPM trajectory readout, reference trajectory load/save, basic analysis (Gaussian fits, orbit comparison, CM step plots).
  • Out: BBA, septum scan, LEDA studies, full automated_startup chain. Future phases.
  • SC dependency: port MATLAB SC calls (SCexp_ALS_*.m, SCgetBPMreading, etc.) to pySC (Python). Reuse pysc-toolkit skill as reference.

Architecture

┌──────────────────┐  CA subscribe   ┌────────────────────────┐
│  Browser (web)   │ ◀─────────────  │  Soft IOC (caproto)    │
│  - Vanilla JS    │                 │  TxT:STATE:*           │
│  - Canvas render │  REST/WS  ◀──▶  │  TxT:RESULT:*          │
│  - CA via WS     │                 │  TxT:CMD:*             │
└──────────────────┘                 └────────────┬───────────┘
                                                  │ caproto client
                                                  ▼
                                     ┌────────────────────────┐
                                     │  ALS control-room PVs  │
                                     │  (BPMs, correctors,    │
                                     │   timing, septum)      │
                                     └────────────────────────┘
  • Backend (Python 3.11+, FastAPI + caproto): single process running both a CA client (reads BPMs/correctors, writes commands) and a soft IOC server (publishes state/results/commands).
  • Frontend (vanilla JS + Canvas, copied from PyBeamViewer pattern): subscribes to result PVs via WebSocket bridge; sends operator commands by writing to TxT:CMD:* PVs.
  • pySC core: trajectory propagation, BPM modeling, response matrix, used both for analysis and for "model vs measurement" overlays.

PV Namespace initial sketch

GroupExamplesPurpose
TxT:CMD:* :ACQUIRE, :LOAD_REF, :RESET Operator commands (write-only from UI)
TxT:STATE:* :STATUS, :N_TURNS, :REF_LOADED App state
TxT:RESULT:BPM:* :X_TBT, :Y_TBT, :RMS_X, :RMS_Y Latest analysis results
TxT:RESULT:FIT:* :KICK_ANGLE, :DISPERSION, :CHI2 Derived quantities
TxT:HEALTH:* :CA_CONNECTED, :LAST_UPDATE Liveness

Tech Stack

  • Python 3.11, FastAPI, uvicorn, caproto (both client & server), pySC, numpy/scipy, Pillow
  • Vanilla JS + Canvas (no framework — same as PyBeamViewer)
  • pytest + Playwright
  • Docker container, deployable to appsdev2 alongside PyBeamViewer (next free port, e.g. 8008)
  • Test IOC isolation: dev IOC uses OSPREY:TEST:TXT:* and ports 59064/59065 per als-profiles safety rules; production uses real TxT:* namespace.

Open Decisions for Kirk to make as he goes

  • WebSocket → CA bridge: roll our own (simple) vs reuse pvws / epics2web (battle-tested). Recommend rolling our own first, keep dep tree small.
  • Whether the soft IOC and FastAPI live in one process (caproto allows this via asyncio) or two services. Recommend single process for simplicity.
  • Auth: skip for v1 (control-room network only, like PyBeamViewer).
