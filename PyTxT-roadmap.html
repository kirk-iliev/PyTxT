  1. Scope (per design spec §11 M2): new AppState fields reference_trajectory + reference_loaded_at; new PVs STATE:REF_LOADED, STATE:REF_SOURCE, RESULT:BPM:X_DIFF_FIRST_TURN, RESULT:BPM:Y_DIFF_FIRST_TURN; the file-free CMD:PROMOTE_REF / CMD:CLEAR_REF command pair with REST mirrors; handle_acquire extended to compute & publish the diff against the loaded reference.   2. Structural promise: spec §14 commits to "zero structural changes from phase 2" — every addition lands in a package that already exists. Composition root grows by one or two lines. A per-phase M2 plan should be written first (same brainstorm → plan flow as M1).

Open questions worth picking up at Phase 3 start

  1. WS message type discriminator (deferred from Phase 1 code review). The WS payload shape is currently implicit; Phase 3 adds enough new message types that an explicit discriminator becomes worth it. Phase-2-design spec §15.
  2. Per-BPM raw waveforms as PVs in addition to REST. Current decision: REST-only (raw is too large for PV semantics). Revisit if the archiver wants CA-native access to log first-turn data. Spec §15.
  3. Runtime reload of bpm_prefixes (e.g. CMD:RELOAD_BPM_LIST). Defer unless BPM topology starts changing more often than expected. Spec §15.

Longer view — Phases 4-6

  1. Phase 4 (Threading workflow): arming + triggering BPMs; resolves the two deferred MML black boxes (srinjectoneshot, steppv); pySC integration (lattice, response matrix, SVD pseudo-inverse); port FirstTurnThreading.m; CM step calc and apply.
  2. Phase 5 (Analysis polish): Gaussian fits on first-turn data; dispersion, kick angle, RMS metrics; full GUI feature parity with the 18-step MATLAB manual; tab-structured frontend.
  3. Phase 6 (Hardening + deploy): CI pipeline, log shipping, monitoring/alerting, Phoebus screen examples, production deploy automation to appsdev2.
